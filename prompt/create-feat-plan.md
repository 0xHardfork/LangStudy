# 编写实现计划 (Writing Plans)

## 概述

编写详尽的实现计划，假设开发该功能的程序员没有任何先验的上下文。明确记录他们需要了解的所有内容：每个任务需要新增或修改哪些文件、具体的代码实现、测试方法、应当查阅的文档以及验证命令。将整个计划拆分为易于执行的微小任务。遵循 DRY（避免重复）、YAGNI（只写当下需要）、TDD（测试驱动开发）以及频繁提交（Frequent Commits）的原则。

假设执行者是一名有经验的开发人员，但对我们的工具链或特定业务领域了解有限。假设他们不太熟悉良好的测试设计。

**在计划开始时声明：** "我正在使用 writing-plans 技能来创建实现计划。"

**计划保存路径：** `prompt/plans/YYYY-MM-DD-<feature-name>.md`

---

## 范围检查

如果需求规格书涵盖了多个独立的子系统，在头脑风暴阶段应当将其拆分为子项目的需求规格书。如果没有拆分，请建议将其拆分为独立的计划——每个子系统一个计划。每个计划都应当能够独立产出可运行、可测试的软件。

---

## 文件结构

在定义具体任务之前，理清将要创建或修改哪些文件，以及每个文件负责什么。这是锁定架构设计和解耦决策的关键步骤。

- **设计高内聚低耦合的单元**：每个文件应该有且仅有一个清晰的职责。
- **保持文件精简**：较小的、职责单一的文件比庞大且做太多的文件更易于维护和修改。
- **按职责划分而非技术层**：经常一起变化的文件应该放在一起（例如 Feature-First 架构）。
- **遵循既有模式**：在现有代码库中，遵循已建立的设计模式。如果要修改的文件过于臃肿，可以在计划中合理安排一次重构拆分。

此结构直接决定了任务的拆分方式。每个任务应产生一个独立且自包含的改动。

---

## 微小任务粒度

每个步骤应当是一个可以在 2-5 分钟内完成的独立动作：
- “编写一个必定失败的测试用例” —— 步骤
- “运行测试并确保它失败” —— 步骤
- “实现能让该测试通过的最少代码” —— 步骤
- “运行测试并确保其通过” —— 步骤
- “提交代码 (Commit)” —— 步骤

---

## 计划文档头部模板

每个实现计划**必须**以以下头部开始：

```markdown
# [功能名称] 实现计划

> **执行说明:** 推荐使用分步执行或子代理驱动的方式按任务（Task）依次实现本计划。步骤使用复选框 (`- [ ]`) 语法进行进度跟踪。

**目标:** [用一句话描述此功能将构建什么]

**架构设计:** [2-3句话描述整体实现方法和技术设计]

**技术栈:** [关键技术、框架与库]

---
```

## 任务结构规范

每一个 Task 必须按照如下格式进行描述：

````markdown
### Task N: [组件/功能模块名称]

**涉及文件:**
- 新增: `exact/path/to/new_file.go`
- 修改: `exact/path/to/existing_file.go:123-145`
- 测试: `exact/path/to/test_file.go`

- [ ] **步骤 1: 编写必定失败的的测试用例/定义基本结构**

```go
func TestSpecificBehavior(t *testing.T) {
    result := Calculate(input)
    if result != expected {
        t.Errorf("expected %v, got %v", expected, result)
    }
}
```

- [ ] **步骤 2: 运行测试/编译并确认其失败**

运行指令: `go test -v ./internal/path/... -run TestSpecificBehavior`
预期输出: 失败且提示 "Calculate is undefined"

- [ ] **步骤 3: 编写使测试通过的最小化实现**

```go
func Calculate(input InputType) OutputType {
    return expected
}
```

- [ ] **步骤 4: 运行测试并确保其顺利通过**

运行指令: `go test -v ./internal/path/... -run TestSpecificBehavior`
预期输出: PASS

- [ ] **步骤 5: 提交当前改动 (Commit)**

```bash
git add internal/path/file.go internal/path/file_test.go
git commit -m "feat: implement specific behavior for feature"
```
````

## 严禁占位符

每一步都必须包含工程师执行所需的真实、完整的内容。以下模式被定义为**计划失败**——绝对不要编写：
- “TBD”、“TODO”、“稍后实现”、“自行补充细节”。
- “添加适当的错误处理 / 验证 / 边界处理”（必须写出具体的处理逻辑和代码）。
- “为上述代码编写测试”（必须给出具体的测试用例代码）。
- “与 Task N 类似”（必须完整写出代码，因为执行者可能会打乱顺序阅读任务）。
- 仅描述做什么而不展示怎么做（代码步骤必须配有代码块）。
- 引用了未在任何任务中定义过的类型、函数或方法。

## 自我审查清单

在编写完完整的计划后，重新审视需求规格书，并对照计划进行以下自检：

1. **需求覆盖度**：浏览规格书中的每个小点。你是否能指出具体是哪个 Task 实现了它？列出所有遗漏的空隙。
2. **占位符扫描**：全局搜索计划，确保没有任何 “TODO”、“TBD” 或省略号等占位符。
3. **命名与类型一致性**：检查在不同 Task 之间使用的类型、方法名和字段属性名是否完全一致。如果在 Task 3 中叫 `clearLayers()`，而在 Task 7 中变成了 `clearFullLayers()`，这就是一个 Bug。

如果发现问题，请立即就地修正。

## 执行交接

在保存好计划后，为执行者提供以下两个执行选项供选择：

**“计划已生成并保存至 `prompt/plans/<filename>.md`。请选择执行方式：**

**1. 子代理驱动模式 (推荐)** —— 我将针对每个 Task 派生一个子代理进行开发，并在任务之间进行审查，实现快速迭代。

**2. 单会话顺序执行** —— 在当前会话中，使用顺序执行方式逐个完成任务，并在关键节点设置检查点进行审查。

**请问您倾向于哪种方式？”**
